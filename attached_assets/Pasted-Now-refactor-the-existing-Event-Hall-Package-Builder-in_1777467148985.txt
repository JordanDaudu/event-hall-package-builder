Now refactor the existing **Event Hall Package Builder** into a real client-facing event hall package configurator.

The project is currently a generic event hall quote builder. The new version should become a Hebrew, RTL, authenticated package-building system for an event hall client.

Main goals:
1. Add authentication and authorization.
2. Add admin-created customer accounts.
3. Add customer identity number support.
4. Add customer-specific base prices.
5. Add customer-specific price overrides for individual package options/upgrades.
6. Add event request details: customer identity number, customer name, phone number, event date, and selected venue.
7. Add support for multiple venues because the event hall business has more than one venue.
8. Replace the generic event type + upgrades flow with a richer interactive event hall package configurator.
9. Redesign the entire frontend to look modern, professional, Hebrew, and right-to-left.
10. Keep a complete backend API that is independent from the frontend.
11. Keep backend-side pricing as the source of truth.
12. Let admins manage inventory/options/prices/customer users/customer-specific prices/venues.
13. Let customers log in, build a package, enter event details, choose a venue, see a live visual preview, and submit the package request to the admin.
14. Make sure submitted and approved quotes keep their original prices and submitted details even if prices, venues, customer details, or options change later.

# Authentication and authorization requirements

Add login and role-based authorization.

Roles:
- `ADMIN`
- `CUSTOMER`

Important business rule:
- There is no public registration.
- Customers cannot create their own accounts.
- Only an admin can create customer accounts.
- The website/system is only given to customers after the hall manager/admin creates an account for them.
- Customer access is approval-based.

Authentication requirements:
- Use Spring Security.
- Use JWT authentication for the React frontend unless the current project structure strongly favors another secure approach.
- Passwords must be securely hashed using BCrypt or Spring Security password encoder.
- Do not store plain text passwords.
- Do not return password hashes in API responses.
- Protect backend endpoints by role.
- Do not rely only on frontend route guards.

Frontend auth requirements:
- The login page is the first screen.
- After login:
  - Admin users are redirected to the admin dashboard.
  - Customer users are redirected to the package builder.
- Add route guards:
  - Anonymous users can only access login.
  - Customers can access customer pages only.
  - Admins can access admin pages only.
- Add Axios token interceptor.
- Handle `401 Unauthorized` by redirecting to login.
- Keep API base URL configurable for Replit/local development.

# Customer account vs event request details

Important distinction:

The customer account is used for login and access control.

The package request must also store the specific event details entered by the customer.

A customer account may have:
- account full name
- customer identity number
- account email
- account phone number
- base package price

But every submitted package request must separately save:
- customer identity number for this event
- contact name for this event
- contact phone number for this event
- event date
- selected venue

Reason:
- The person logging in might be the customer, parent, partner, or event organizer.
- The phone number for this event may be different from the login account phone number.
- The identity number should be snapshotted for the submitted request.
- The customer may submit more than one event request in the future.
- The business has multiple venues, so each request must say which venue the customer chose.
- Old submitted requests must preserve the event details exactly as submitted.

Recommended frontend behavior:
- In the package builder, add an initial step called:

```text
פרטי אירוע
```

This step should ask for:
- שם מלא
- מספר תעודת זהות
- מספר טלפון
- תאריך האירוע
- בחירת אולם

The form may prefill name, identity number, and phone from the logged-in user account, but the customer should be able to edit them before submitting the package request if the business wants that flexibility.

Backend rule:
- Do not rely only on the logged-in user account fields.
- Save the request-specific customer identity number, contact name, and phone number directly on the submitted package request.

# Critical pricing requirement: customer-specific base price

The base package price changes from customer to customer.

Business rule:
- When the admin creates a customer, the admin assigns a personal base package price.
- When the customer logs in and builds a package, the total starts from that customer’s personal base package price.
- The frontend can show a live estimate, but the backend must calculate the final total.
- The backend must snapshot the customer’s base package price into the submitted package request.

Example:
- Customer A has a base package price of ₪20,000.
- Customer B has a base package price of ₪25,000.
- Both may select the same options, but their final totals start from different base prices.

Important:
- Do not accept `basePackagePrice` from the frontend when a customer submits a package request.
- The backend must load it from the authenticated customer account.
- The customer identity must come from the JWT/security context.

# Critical pricing requirement: customer-specific option price overrides

Admins must be able to override prices per customer for specific package options/upgrades.

Example:
- The default price of a premium chuppah is ₪2,000.
- For Customer A, the admin may set this same chuppah to ₪1,500.
- For Customer B, the same chuppah may remain ₪2,000.
- For Customer C, the admin may set this same chuppah to ₪0 if it was included in their deal.

This means pricing is not only global.

There are two pricing levels:

1. Global option price:
   - Stored on `PackageOption.price`.
   - Used by default for all customers.

2. Customer-specific option price override:
   - Used only for one specific customer.
   - If an active override exists for the customer and option, it replaces the global option price.

Recommended entity:

## `CustomerOptionPriceOverride`

Fields:
- `id`
- `customer`
- `option`
- `overridePrice`
- `active`
- `notes`
- `createdAt`
- `updatedAt`

Database rule:
- There should be only one active override per customer per option.
- Add a unique constraint if possible on:
  - `customer_id`
  - `option_id`

Pricing priority:

```text
If customer has an active price override for the selected option:
    use customer-specific override price
else:
    use global option price
```

Important:
- An override price is not a discount.
- It is the actual price that should be used for that customer.
- The override price can be:
  - higher than the global price
  - lower than the global price
  - zero
- Admin should be able to enable/disable overrides without deleting pricing history.

Recommended admin API for customer-specific prices:

```text
GET    /api/admin/customers/{customerId}/price-overrides
POST   /api/admin/customers/{customerId}/price-overrides
PUT    /api/admin/customers/{customerId}/price-overrides/{overrideId}
PUT    /api/admin/customers/{customerId}/price-overrides/{overrideId}/active
DELETE /api/admin/customers/{customerId}/price-overrides/{overrideId}
```

Suggested request DTO:

```json

  "optionId": 123,
  "overridePrice": 1500,
  "active": true,
  "notes": "Special price agreed with this customer"
}
```

# Critical pricing requirement: quote price snapshots and approved quote locking

All submitted package requests must snapshot prices and event details.

This is critical because:
- Global option prices can change later.
- Customer-specific option prices can change later.
- Customer base package prices can change later.
- Customer identity number/account details can change later.
- Benefit rules can change later.
- Inventory/options can be enabled or disabled later.
- Venue names/details can change later.

Old submitted quotes must not change because of future changes.

When a customer submits a package request:
- Save the customer’s base package price at that moment into `basePackagePriceSnapshot`.
- Save the request-specific customer/contact details:
  - `eventCustomerIdentityNumber`
  - `eventContactName`
  - `eventContactPhoneNumber`
  - `eventDate`
- Save the selected venue.
- Also snapshot the selected venue name into:
  - `venueNameSnapshot`
- For every selected option, save:
  - the global option price at the time
  - the customer-specific override price at the time, if used
  - the actual price used before benefits
  - the final price after benefits
  - whether a benefit was applied
  - the benefit reason, if any

Recommended `PackageRequest` / `Quote` fields:
- `id`
- `customer`
- `eventCustomerIdentityNumber`
- `eventContactName`
- `eventContactPhoneNumber`
- `eventDate`
- `venue`
- `venueNameSnapshot`
- `status`
- `basePackagePriceSnapshot`
- `totalPrice`
- `createdAt`
- `submittedAt`
- `approvedAt`
- `summaryNotes`
- `knightTableCount`
- selected items collection

Recommended `PackageRequestItem` / `QuoteItem` price fields:
- `globalPriceSnapshot`
- `customerOverridePriceSnapshot`
- `priceAtSelection`
- `finalPrice`
- `benefitApplied`
- `benefitReason`

Approved quote rule:
When a quote/package request status changes to `APPROVED`, it becomes price-locked.

After a quote is approved:
- Future changes to global option prices must not affect it.
- Future changes to customer-specific override prices must not affect it.
- Future changes to the customer base package price must not affect it.
- Future changes to the customer account identity number must not affect the submitted request snapshot.
- Future changes to benefit rules must not affect it.
- Future option enable/disable changes must not affect it.
- Future venue name/details changes must not affect the displayed submitted venue name.
- The quote must continue displaying the snapshotted prices and event details saved at submission/approval time.

Recommended behavior:
- Submitted quotes already store price snapshots.
- Approved quotes must never be recalculated from current database prices.
- Quote detail pages should read from saved snapshots, not from current option prices.
- Do not recalculate approved quote totals from current `PackageOption.price`.

Optional stricter rule:
After a quote is `APPROVED`, prevent editing selected items or recalculating totals unless a future explicit “revision” feature is created.

If revision support is added later:
- Create a new quote version.
- Do not overwrite the accepted quote.

# Table requirements

There are two table contexts:
1. Regular table
2. Knight table / large table

Business flow:
- Customer first decides whether to include knight tables.
- Knight table count must be between 0 and 4.
- If knight table count is 0:
  - no knight table design is required.
- If knight table count is at least 1:
  - customer must design regular tables
  - customer must also design knight tables separately

Regular table design is always required.

Table design steps:

## Step 1: Frame selection

Requirements:
- Customer chooses a table frame.
- Frame options can include:
  - free options
  - paid options
  - “no frame” option
- “No frame” can also have a price if the business wants that.

## Step 2: Flower selection

Flower options must support:
- Categories
- Flower size:
  - large flower
  - small flower

Flower selection rules:
- Primary flower selection is required.
- Primary flower must be exactly one option:
  - either one large flower
  - or one small flower
- If primary flower is large:
  - customer may optionally choose one secondary small flower.
- If primary flower is small:
  - customer cannot choose a secondary small flower.

Examples:
- Large flower only: valid.
- Large flower + small flower: valid.
- Small flower only: valid.
- Small flower + secondary small flower: invalid.

## Step 3: Candle holder selection

Candle holders can be selected in two modes:

1. Random mode:
   - The hall decides and styles the candle holders.
   - Customer does not choose specific candle holders.

2. Selected mode:
   - Customer chooses between 1 and 3 candle holder options.

Rules:
- Candle holder options can be free or paid.
- If mode is `RANDOM`, selected candle holder list must be empty.
- If mode is `SELECTED`, customer must select 1 to 3 candle holder options.
- Backend must validate this.

# Backend pricing service

Create or update a dedicated service such as `PackagePricingService`.

It should:
1. Load the authenticated customer.
2. Validate that the customer is active.
3. Validate event request details.
4. Validate selected venue.
5. Start with `customer.basePackagePrice`.
6. Snapshot `basePackagePrice` into the package request.
7. Snapshot event customer identity number, contact name, phone number, event date, and venue name.
8. Validate all selected options.
9. Validate compatibility rules.
10. For every selected option:
    - check if the customer has an active price override
    - if yes, use the customer-specific override price
    - if not, use the global option price
11. Save the global option price snapshot.
12. Save the customer override price snapshot if used.
13. Save the actual price used before benefits.
14. Apply benefit rules such as free tablecloth.
15. Save the final price after benefits.
16. Save benefit reason if a benefit was applied.
17. Save final total into the package request.
18. Never recalculate approved quotes from current prices.

Pricing formula:

```text
total =
    customer.basePackagePrice snapshot
    + sum(final price of all selected package request items)
```

Where each item’s base price is resolved as:

```text
if active customer-specific override exists:
    item price = override price
else:
    item price = global package option price
```

Then benefit rules may modify final item price.

Example:
- Global tablecloth price: ₪500
- Customer override price: ₪300
- Benefit rule makes it free because of selected chuppah
- `globalPriceSnapshot = 500`
- `customerOverridePriceSnapshot = 300`
- `priceAtSelection = 300`
- `finalPrice = 0`
- `benefitApplied = true`
- `benefitReason = "Included because of selected chuppah"`
